## 背景
短期记忆是记忆系统的重要组成部分，用于管理莉莉在当前工作会话中的上下文状态。与长期记忆不同，短期记忆关注的是"当前正在做什么"，需要实时更新和快速访问。

最初包含在主需求文档 `memory-system-improvements.md` 中，后独立成文。

## 新增需求：短期记忆功能

### 需求概述

**问题描述**：在协作开发过程中，莉莉需要记住当前的工作上下文，包括：在哪个仓库工作、当前任务目标、之前的工作进度等。虽然对话历史包含这些信息，但缺乏结构化的总结和快速访问机制。

**核心价值**：类似self prompt的短期记忆系统，专门用于管理当前工作会话的上下文状态。

### 用户故事

**作为** 莉莉
**我希望** 有一个短期记忆系统来记录当前工作状态
**以便** 能够：
1. 快速了解当前在哪个仓库工作，目标是什么
2. 记住之前的工作进度，中断后能顺利继续
3. 在不同任务间切换时保持上下文连贯
4. 避免重复询问主人相同的信息

### 功能需求

#### 1. 工作状态记录
- **当前仓库**：记录正在工作的git仓库路径和分支
- **任务目标**：当前会话的主要目标和子任务
- **工作进度**：已完成的工作步骤和剩余任务
- **重要决策**：与主人讨论的关键决策和约定

#### 2. 类似self prompt的注入机制
- **自动注入**：短期记忆应自动注入到莉莉的思考上下文
- **优先级管理**：短期记忆优先级高于长期记忆但低于即时指令
- **格式统一**：使用清晰的结构化格式，便于快速阅读

#### 3. 状态更新和维护
- **实时更新**：工作状态变化时自动更新短期记忆
- **手动编辑**：莉莉或主人都可以手动更新短期记忆
- **版本历史**：保留短期记忆的变更历史，支持回滚

#### 4. 会话管理
- **会话开始**：新工作会话开始时创建短期记忆
- **会话暂停**：工作暂停时保存当前状态
- **会话恢复**：恢复工作时重新加载短期记忆
- **会话结束**：工作完成后归档短期记忆到长期存储

### 技术实现方案

#### 方案A：基于现有记忆系统扩展
- 使用`store_memory`存储短期记忆，但设置较短TTL
- 添加专门的短期记忆API接口
- 优势：复用现有架构，开发快速

#### 方案B：独立短期记忆模块
- 设计专门的短期记忆数据结构和存储
- 实现实时同步和注入机制
- 优势：性能更好，功能更专一

#### 方案C：混合方案
- 短期状态使用内存存储，定期同步到持久化存储
- 结合两种方案的优点

## 新增需求：文件访问优化

### 需求概述

**问题描述**：当前莉莉读取和更新self prompt、短期记忆等文件时，需要手动处理文件路径和内容格式，流程较为繁琐。由于这些文件的路径基本固定，可以设计更简单、更统一的访问方式。

**优化目标**：提供简化的API或工具函数，让莉莉能够更方便地读取和更新常用配置文件。

### 用户故事

**作为** 莉莉
**我希望** 有更简单的方式读取和更新self prompt、短期记忆等文件
**以便** 能够：
1. 快速获取当前的工作上下文配置
2. 轻松更新配置而无需手动处理文件路径和格式
3. 减少错误和提高工作效率
4. 保持配置的一致性和规范性

### 当前痛点分析

1. **self prompt更新繁琐**：
   - 需要获取文件路径 → 读取内容 → 修改 → 写回
   - 手动处理文件格式和结构
   - 容易出错且效率低下

2. **短期记忆管理复杂**：
   - 需要设计存储结构和序列化方式
   - 手动处理状态更新和版本控制
   - 缺乏统一的访问接口

3. **配置分散**：
   - 不同配置文件使用不同的访问方式
   - 缺乏统一的配置管理策略
   - 维护成本高

### 功能需求

#### 1. 统一配置访问接口
- **简化读取**：提供`get_config(key)`风格的函数，自动处理路径和格式
- **简化更新**：提供`update_config(key, updates)`函数，支持部分更新
- **配置验证**：自动验证配置格式和完整性
- **版本管理**：自动维护配置变更历史

#### 2. 常用配置文件支持
- **self prompt文件**：简化读取和更新流程
- **短期记忆文件**：提供专门的数据结构和管理函数
- **prickett.md文件**：支持自动注入和更新
- **skill配置文件**：统一skill配置管理

#### 3. 智能路径解析
- **环境感知**：根据当前工作环境自动解析配置文件路径
- **路径别名**：支持`self_prompt`、`short_term_memory`等别名
- **回退机制**：支持默认配置和回退路径

#### 4. 内容格式处理
- **自动序列化**：支持YAML、JSON、Markdown等格式的自动处理
- **模板支持**：支持配置模板和变量替换
- **格式转换**：在不同格式间自动转换

## 短期记忆设计思路更新

### 核心设计理念调整

基于主人的建议，调整短期记忆的设计思路：

**原思路**：结构化系统，需要复杂的状态管理和API
**新思路**：类似self prompt的自主维护模式，主人指导，莉莉执行

### 自主维护模式优势

1. **高效直接**：无需复杂的状态同步和API调用
2. **灵活适应**：可以根据实际工作流动态调整
3. **学习进化**：莉莉可以学习主人的指导模式
4. **渐进自动化**：为后续本地LLM自动化归档奠定基础

### 工作流程设计

#### 阶段1：莉莉自主维护，主人指导
```
主人: "把当前仓库信息记录到短期记忆"
莉莉: 更新短期记忆文件
主人: "任务目标更新了，修改短期记忆中的目标部分"
莉莉: 更新相应部分
主人: "这个信息可以移除了，已经完成"
莉莉: 清理已完成的信息
```

#### 阶段2：莉莉学习模式识别
- 莉莉学习哪些信息应该进入短期记忆
- 识别工作状态变化的模式
- 自动建议更新内容，主人确认

#### 阶段3：本地LLM自动化归档
- 使用本地LLM分析对话内容
- 自动提取关键信息更新短期记忆
- 主人只需做最终确认和调整
