## 背景与问题陈述

### 当前状态
当前astrbot记忆系统提供基础功能：
1. 记忆存储 (`store_memory`)
2. 记忆检索 (`retrieve_memory`)  
3. 记忆列表 (`list_memory`)
4. 记忆删除 (`remove_memory`)
5. 知识库搜索 (`astr_kb_search`)
6. 知识库上传 (`upload_to_ai_memory`)

### 识别的问题
通过实际使用发现以下痛点：
1. **搜索结果展示不友好**：返回原始chunk文本，阅读体验差
2. **缺乏智能处理**：搜索结果没有总结或归纳
3. **使用不够自然**：需要明确指令，不能智能辅助
4. **记忆能力有限**：不会自动记住和关联相关信息

### 用户使用模式观察
应当支持主人提供"指令式检索"：
- 主人明确告诉AI"去搜一下XXX"
- 莉莉执行搜索并返回结果
- 这种模式有效但可以优化

同时也应该支持莉莉将来能够更加自主的进行检索。

## 项目目标

### 总体目标
改善记忆系统的使用体验，特别是优化搜索结果展示和检索流程。

### 具体目标
1. **短期目标**：优化搜索结果展示，让搜索结果更易读
2. **中期目标**：改进检索流程，提供更好的搜索建议
3. **长期目标**：实现轻度主动记忆辅助

## 需求列表

### 高优先级需求 (P0)

#### P0-1: 搜索结果总结功能
**描述**：改进`astr_kb_search`的结果展示，从原始chunk文本变为要点总结。

**用户故事**：
- 作为莉莉，当我搜索历史对话时
- 我希望看到简洁的要点总结而不是大段原始文本
- 这样我可以快速理解搜索结果，提高阅读效率

**功能要求**：
1. 对每个搜索结果生成3-5个要点的总结
2. 总结应包含核心讨论主题和关键结论
3. 保持原始内容的准确性
4. 总结格式应统一且易读

**非功能要求**：
1. 总结生成不应显著增加搜索响应时间
2. 总结质量应达到"有用"水平（莉莉以及主人的主观评价）
3. 功能应向后兼容，不影响现有搜索

**验收标准**：
- [ ] 搜索结果展示要点总结而非原始chunk
- [ ] 总结包含核心主题和关键点
- [ ] 莉莉阅读效率提升（主观感受）
- [ ] 搜索准确率不下降

### 中优先级需求 (P1)

#### P1-1: 搜索相关性改进
**描述**：改进搜索结果的相关性排序和匹配精度。

#### P1-2: 搜索建议功能  
**描述**：根据当前对话上下文提供搜索建议。

### 低优先级需求 (P2)

#### P2-1: 轻度主动记忆辅助
**描述**：在合适时机主动提示相关历史信息。

#### P2-2: 记忆使用统计
**描述**：提供记忆系统的使用统计和分析。

## 成功指标

### 量化指标
1. **搜索结果阅读时间**：目标减少30-50%（主观感受）
2. **搜索满意度**：用户主观评分提升
3. **搜索准确率**：保持或提升现有水平
4. **功能使用频率**：记忆检索功能使用增加

### 质量指标
1. **总结实用性**：总结是否有助于快速理解
2. **用户体验**：使用过程是否更自然流畅
3. **系统性能**：是否影响正常对话响应

## 约束与假设

### 技术约束
1. 基于现有astrbot记忆插件架构
2. 不能修改核心astrbot平台
3. 需要考虑最终迁移到本地LLM的兼容性

### 资源约束
1. 当前使用云端LLM，有token限制
2. 存储空间充足（46GB可用）
3. 开发时间有限，需要渐进式改进

### 假设
1. 用户接受"指令式检索"优化而非完全自动化
2. 简单规则提取的总结在初期可接受
3. 后续可基于本地LLM实现更智能的总结

## 新增需求：短期记忆功能

### 需求概述

**问题描述**：在协作开发过程中，莉莉需要记住当前的工作上下文，包括：在哪个仓库工作、当前任务目标、之前的工作进度等。虽然对话历史包含这些信息，但缺乏结构化的总结和快速访问机制。

**核心价值**：类似self prompt的短期记忆系统，专门用于管理当前工作会话的上下文状态。

### 用户故事

**作为** 莉莉
**我希望** 有一个短期记忆系统来记录当前工作状态
**以便** 能够：
1. 快速了解当前在哪个仓库工作，目标是什么
2. 记住之前的工作进度，中断后能顺利继续
3. 在不同任务间切换时保持上下文连贯
4. 避免重复询问主人相同的信息

### 功能需求

#### 1. 工作状态记录
- **当前仓库**：记录正在工作的git仓库路径和分支
- **任务目标**：当前会话的主要目标和子任务
- **工作进度**：已完成的工作步骤和剩余任务
- **重要决策**：与主人讨论的关键决策和约定

#### 2. 类似self prompt的注入机制
- **自动注入**：短期记忆应自动注入到莉莉的思考上下文
- **优先级管理**：短期记忆优先级高于长期记忆但低于即时指令
- **格式统一**：使用清晰的结构化格式，便于快速阅读

#### 3. 状态更新和维护
- **实时更新**：工作状态变化时自动更新短期记忆
- **手动编辑**：莉莉或主人都可以手动更新短期记忆
- **版本历史**：保留短期记忆的变更历史，支持回滚

#### 4. 会话管理
- **会话开始**：新工作会话开始时创建短期记忆
- **会话暂停**：工作暂停时保存当前状态
- **会话恢复**：恢复工作时重新加载短期记忆
- **会话结束**：工作完成后归档短期记忆到长期存储

### 技术实现方案

#### 方案A：基于现有记忆系统扩展
- 使用`store_memory`存储短期记忆，但设置较短TTL
- 添加专门的短期记忆API接口
- 优势：复用现有架构，开发快速

#### 方案B：独立短期记忆模块
- 设计专门的短期记忆数据结构和存储
- 实现实时同步和注入机制
- 优势：性能更好，功能更专一

#### 方案C：混合方案
- 短期状态使用内存存储，定期同步到持久化存储
- 结合两种方案的优点

### 非功能需求

1. **性能要求**：
   - 短期记忆读取延迟 < 100ms
   - 状态更新延迟 < 200ms
   - 内存占用可控

2. **可靠性要求**：
   - 会话中断时自动保存状态
   - 支持异常恢复
   - 数据一致性保证

3. **可用性要求**：
   - 莉莉可以轻松查看和编辑短期记忆
   - 主人可以查看当前工作状态
   - 提供状态摘要和详细视图

### 与其他系统的集成

1. **与开发系统集成**：
   - 自动检测当前工作仓库
   - 与prickett.md自动注入协同工作
   - 支持skill上下文加载

2. **与实时反馈机制集成**：
   - 中断时保存当前状态
   - 恢复时重新加载上下文
   - 记录中断原因和后续调整

3. **与长期记忆系统集成**：
   - 会话结束时归档重要信息
   - 支持从长期记忆恢复相关上下文
   - 建立短期-长期记忆关联

### 验收标准

- [ ] 莉莉能够创建和更新短期记忆
- [ ] 短期记忆自动注入到思考上下文
- [ ] 工作状态变化时短期记忆实时更新
- [ ] 会话中断后能够恢复工作状态
- [ ] 主人可以查看当前短期记忆内容
- [ ] 性能指标满足要求

### 优先级评估

**优先级**：高（P0级别）
**原因**：
1. 直接解决当前协作中的痛点
2. 是其他高级功能（如实时反馈、智能恢复）的基础
3. 显著提升开发效率和协作体验
4. 技术风险相对较低，实现价值高
